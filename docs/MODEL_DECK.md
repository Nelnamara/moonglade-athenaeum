### Methods — Nel Key-Art
- **Artist-tag stacking on Illustrious/NoobAI (the core technique for this camp)** — 1) Use EXACT Danbooru artist tag names (underscores, escaped parens like arknights artist tags: name_\(circle\)) - 'we run tags how they are shown on Danbooru'. Find them at danbooru.donmai.us/artists. 2) An artist must have enough Danbooru posts to be learned - ~20 posts is NOT enough (use a style LoRA instead); aim for artists with 100+ posts. 3) Stack 2-4 artists (tooling like EasyNoobai caps at 5): one 'structure' artist at full weight for composition + 1-2 'flavor' artists down-weighted. Weight syntax is standard A1111/Forge attention: (artist_name:0.8), and on NoobAI-flavored models (artist:name:0.5) also works. Swapping a single artist tag shifts the whole tone (e.g. kazuki_yone -> greenteaa goes ethereal -> cinematic) without changing content. 4) Position matters: earlier in the prompt = stronger; on RouWei/JANKU use the 'by artistname' prefix and isolate the artist block in its own CLIP chunk with BREAK (mandatory on RouWei when mixing several). 5) Fewer, cleaner tags beat dozens stacked - style dilutes past the ~75-token chunk boundary. 6) TO DISCOVER which artists a model knows: the Illustrious & NoobAI Style Explorer indexes 16,000+ artist styles with preview grids, Danbooru dataset-strength data, and one-click prompts (works for Forge); plus the '480 Artist Tags NoobAI Comparative Study' article for side-by-side grids. ### Methods — Lifelike Nel
- **Lavender skin without bleed on Illustrious/tag models: 'colored skin, purple skin' + explicit colors everywhere else + targeted negatives** — Danbooru's tag system defines 'colored skin' as the umbrella for unnatural skin tones, with 'purple skin' as a specific child tag that implicates it — meaning every purple-skinned training image carried BOTH tags in its caption, so prompt with both together: 'colored skin, purple skin' (add 'pale skin' to lighten toward lavender rather than deep violet). Place these tokens early and adjacent to the character tokens (1girl, elf, pointy ears, long ears). Then defend against bleed from both directions: (a) explicitly color every other major element ('silver hair, glowing eyes, green eyes') — any element left colorless inherits the dominant color token; (b) negative-prompt the specific bleed targets you don't want: 'purple hair, purple eyes, purple background, purple clothes'. Keep CFG moderate (4-6 on these semi-real merges) — high CFG amplifies color contamination. A dedicated 'Colored Skin' LoHa exists on Civitai (model 19695) but it is SD1.5-era — dated; prompt technique alone is usually enough on Illustrious, which saw plenty of purple_skin posts (dark elves, shinobu, Draenei/night-elf fanart).
  - sources: https://danbooru.donmai.us/wiki_pages/colored_skin · https://moescape.ai/posts/176c4842-561e-49cc-aa02-4c0036100c8e · https://civitai.com/models/19695/colored-skin-conceptanime
- **Lavender skin on Chroma/Flux (prose models): bind the color to the noun, restate it once, name the lighting** — Chroma's T5-based prose encoder binds adjectives to adjacent nouns far better than CLIP-tag models, so the trick is placement, not tags: put the color directly against 'skin' ('her skin is a soft pale lavender-violet, smooth and matte') and then restate it once in a different clause later in the prompt ('lavender skin catching the cool moonlight') — a single mention can get averaged away in long prompts. Explicitly describe the light source color too ('cool silver-blue moonlight', 'warm gold rim light') because on non-human skin tones the model otherwise 'corrects' skin toward human under neutral light. Name every other element's color in prose exactly as with the tag models. This mirrors the user's existing hex-per-element LOCK-block style — that discipline is precisely what prevents bleed here; Chroma runs it at guidance ~3-4 where color contamination is naturally low.
  - sources: https://huggingface.co/lodestones/Chroma1-HD
- **Keep the character recognizable across realism levels: anime-gen then img2img refine in the semi-real merge** — The most reliable identity-preserving pipeline on this hardware: generate the character in the already-dialed WAI-NSFW-Illustrious (where her tags/LoRAs behave), then img2img that output in Illustrij/Equinox/Semi-Real MM at denoise 0.35-0.55 — low enough to keep face structure, skin tone, and costume, high enough to let the semi-real merge repaint skin texture, lighting, and depth. Because all three recommended merges share the Illustrious base, any character LoRA that works in WAI works in them at the same weight (the user's base-model-version rule holds — these are all safe). Steer realism per-image with the 'realistic' positive tag (Equinox is documented as steerable this way) rather than cranking LoRA weights, which the user has already learned destroys composition.
  - sources: https://civitai.com/models/1833737/bss-equinox-il-semi-realistic-model · https://civitai.com/models/1025051/illustrij

### Methods — Character-LoRA Training
- **Verdict: train Illustrious locally, rent cloud for Flux 2** — On a 4070 Super 12GB, SDXL/Illustrious character LoRA training is comfortably solved (kohya_ss or OneTrainer) and the result runs in Forge with WAI-NSFW-Illustrious. So: build the character LoRA on Illustrious now, and if you later want her in Maestro, do a one-off ai-toolkit cloud run (RunComfy/RunPod) — a few dollars, 1-3 hours.
  - sources: https://www.runcomfy.com/trainer/ai-toolkit/flux-2-klein-lora-training · https://civitai.com/articles/10872/training-sdxl-lora-in-kohyass-with-8gb-12gb-or-higher-vram · https://github.com/cocktailpeanut/fluxgym
- **Dataset curation from 'hundreds of images' (sweet spot 50-150)** — More is not better past ~150 — curate, don't dump. Target: 10-20 clean face/portrait shots + 20-40 half-body/full-body, maximizing variety in angle, expression, outfit, lighting, background (variety prevents a rigid LoRA; near-duplicates cause overbaking). Since many of your PixAI gens are 2x2 grids, crop to singles first. Drop images with mangled hands/extra limbs — the LoRA learns defects too. Use repeats to hit ~1600-3000 total steps (e.g., 100 images x 5 repeats x 1 epoch ≈ 500 seen-images per the DCAI recipe, or 40 images x 10 repeats x ~5 epochs). Train on base Illustrious-XL (v1/v2) rather than the WAI merge if you want the LoRA portable across Illustrious finetunes; train directly on WAI-NSFW-Illustrious v11 if you only care about your Forge setup — both are common practice.
  - sources: https://www.digitalcreativeai.net/en/post/original-character-lora-illustrious-character-training · https://civitai.com/articles/5255/my-sdxl-lora-experience-12gb-rtx3060-gpu-local-training-wkohya-ss · https://diffusiondoodles.substack.com/p/how-to-train-a-lora
- **Captioning: Danbooru tags + trigger word, prune identity tags** — For Illustrious character LoRAs use Danbooru-style tags (WD14/wd-eva02 auto-tagger, built into OneTrainer; kohya has it under Utilities), matching how the base was trained — this aligns with your learned rule 'Illustrious wants Danbooru tags for characters.' Structure: trigger word first (e.g., 'nelnamara'), then tags. Critical trick: DELETE the tags that define her identity (lavender/purple skin, pointy ears, night elf, her hair color/markings) so those features absorb into the trigger word; KEEP tags for things you want to stay variable (outfit, pose, expression, background). Tag it and it stays promptable; untag it and it bakes in. Note the DCAI guide instead uses pure natural-language captions on Illustrious-XL v2 — that also works (v2 handles prose), but tag-based is the safer default for a character on WAI/Illustrious v1-era merges.
  - sources: https://www.digitalcreativeai.net/en/post/original-character-lora-illustrious-character-training · https://civitai.com/articles/24648/lora-training-guide-sdxl-or-pony-or-illustrious-kohyass · https://github.com/bmaltais/kohya_ss/discussions/3044
- **12GB training settings + time estimate (Illustrious/SDXL)** — Network: dim 16-32, alpha = dim/2 (dim 8/alpha 1 works per DCAI and gives tiny files; dim 128 is overkill for one character and slower — published 12GB guides range 8-128, character consensus sits 16-32). LR 1e-4 UNet / 5e-5 text-encoder, AdamW8bit, cosine w/ 10% warmup (or Prodigy at LR 1.0 to skip LR tuning). Batch 1 (2 if it fits), 1024px + aspect bucketing, bf16, gradient checkpointing ON, cache latents + text-encoder outputs, clip_skip 2. Steps: 1600-3000 total; save checkpoints every epoch and test the last 3-4 — best is often not the final one. Time at 12GB: RTX 3060 12GB reports 3-5h; RTX 3090 does ~1500 steps in 30-60 min; the 4070 Super sits much closer to the 3090, so expect roughly 1-2.5 hours per run — iterate freely.
  - sources: https://www.digitalcreativeai.net/en/post/original-character-lora-illustrious-character-training · https://civitai.com/articles/10872/training-sdxl-lora-in-kohyass-with-8gb-12gb-or-higher-vram · https://civitai.com/articles/5255/my-sdxl-lora-experience-12gb-rtx3060-gpu-local-training-wkohya-ss
- **Flux LoRA at 12GB in 2026: FLUX.1 yes, Flux 2 / Chroma no** — Quantized 12GB Flux training exists ONLY for the FLUX.1 generation: FluxGym (fp8 base, Kohya backend) has an explicit 12GB preset, ~6-10h for 20 images — proven but dated, and the output LoRA is useless in Maestro's Flux 2 Klein 9B (different architecture) and generally incompatible with Chroma (pruned/modified schnell — Chroma needs LoRAs trained on Chroma itself, which ai-toolkit supports but with no verified 12GB recipe).
